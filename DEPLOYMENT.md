# Deploying Cult Records on Render

This guide describes how to reproduce the Cult Records deployment from a new Render account using the Render Dashboard. The repository already contains the production configuration in `render.yaml`, so the recommended approach is to deploy it as a Render Blueprint rather than creating the web service and database separately.

The current deployment uses:

| Resource | Configuration |
| --- | --- |
| Source repository | `https://github.com/chriskalos/cult-records` |
| Git branch | `main` |
| Render web service | `cult-records` |
| Render database | `cult-records-db` |
| Image storage and delivery | Cloudinary |
| Region | Frankfurt |
| Web service plan | Free |
| Database plan | Free PostgreSQL |
| Default Render address | `https://cult-records.onrender.com` |
| Custom address | `https://cult.chriskalos.xyz` |
| DNS provider | Cloudflare |

The repository is also mirrored to Tangled. That mirror is useful as another copy of the source, but Render's Git-backed deployment connects to GitHub. A change must reach the linked GitHub branch before Render can deploy it automatically.

## 1. Prepare the accounts and repository

You need:

- A GitHub account with access to the Cult Records repository, or a fork of it.
- A Render account.
- A Cloudinary account and its `CLOUDINARY_URL` credential.
- A Cloudflare account that manages the intended custom domain, if a custom domain is required.
- A Stripe account with a sandbox, only if checkout needs to work.

Open the repository on GitHub and confirm that `render.yaml`, `build.sh`, `requirements.txt`, and the Django source are present on the `main` branch. Render reads `render.yaml` from the repository root.

If deploying an independent copy, fork the GitHub repository first. Before creating the Blueprint, update these values in the fork's `render.yaml`:

- Change the service name if `cult-records` is already taken.
- Change or remove `cult.chriskalos.xyz` under `domains`.
- Change or remove the matching `CUSTOM_DOMAIN` value.

Removing the custom-domain entries is the simplest option when a domain has not been chosen yet. The application will still work at its generated `onrender.com` address. Add the domain to the Blueprint and Django configuration later when the domain is available.

Do not place database passwords, Django secret keys, Cloudinary credentials, Stripe keys, or webhook secrets in GitHub. The Blueprint generates, links, or prompts for the required values inside Render.

## 2. Connect GitHub to Render

1. Sign in to the [Render Dashboard](https://dashboard.render.com/).
2. Open the **New** menu in the upper-right corner and select **Blueprint**.
3. Choose **GitHub** when Render asks for a Git provider.
4. GitHub opens an authorization page for the Render GitHub app. Grant access to the Cult Records repository. Selecting only this repository is sufficient.
5. Return to Render. Search for the Cult Records repository and select it.
6. Select the `main` branch if Render asks for a branch.
7. Leave the Blueprint path as `render.yaml`, because the file is at the repository root.

If the repository does not appear, open the Render GitHub app settings on GitHub and add the repository to its allowed repository list. Then refresh the repository list in Render.

## 3. Review and deploy the Blueprint

Render reads `render.yaml` and presents the resources it intends to create. Confirm that the preview contains both of these resources:

- A Python web service named `cult-records` on the Free plan in Frankfurt.
- A PostgreSQL database named `cult-records-db` on the Free plan in Frankfurt.

The Blueprint configures the web service with the following behavior:

- `./build.sh` is the build command.
- Gunicorn starts Django and listens on Render's assigned port.
- `/` is the health-check path.
- Deployments occur automatically after changes reach the linked GitHub branch.
- `DATABASE_URL` comes from the PostgreSQL database's internal connection string.
- `SECRET_KEY` is generated by Render.
- `CLOUDINARY_URL` is supplied securely through Render and is never stored in the repository.
- Existing version-controlled artwork and future admin uploads are delivered through Cloudinary.
- Django debug mode is disabled.
- The Render hostname and configured custom hostname are accepted by Django.
- WhiteNoise serves the collected static files.

Render displays a field for `CLOUDINARY_URL` because it is deliberately marked as a value that is not stored in the repository. Enter the Cloudinary API environment variable for the intended product environment. The value starts with `cloudinary://` and is available from the Cloudinary Console. Do not paste it into `render.yaml`.

Render might also display fields for `STRIPE_SECRET_KEY` and `STRIPE_WEBHOOK_SECRET`. Leave them empty if checkout is not needed. They can be added later from the service's **Environment** page.

For an existing Blueprint, `sync: false` does not prompt again when `render.yaml` changes. Open the web service's **Environment** page, add `CLOUDINARY_URL` manually, and save it before deploying this version. The application deliberately refuses to start on Render without this variable so uploads cannot fall back to the ephemeral filesystem.

Review the proposed changes, then select **Deploy Blueprint**. Render begins provisioning the database and building the web service.

## 4. Follow the first deployment

Open the new `cult-records` web service from the Blueprint page. The **Events** and **Logs** views show the build and deployment progress.

During the build, `build.sh` performs four tasks:

1. Installs the pinned packages from `requirements.txt`.
2. Collects Django's static files for WhiteNoise.
3. Applies all Django database migrations to PostgreSQL.
4. Verifies legacy uploaded media in Cloudinary, copies any files that remain available from the previous live service, and clears references to legacy files that already return 404.

The first database is new. It does not copy a developer's local SQLite file. Data created by the project's migrations, including demonstration data, is added automatically. Any other local data requires a separate export and import process.

Wait until the latest event is marked **Live**. Open the service's `onrender.com` address and confirm that the home page loads over HTTPS. The first request after a period of inactivity can take around a minute because a free web service spins down when idle.

If the deployment fails, open **Logs** and inspect the first error rather than repeatedly redeploying. Common causes are a missing repository file, a dependency installation error, or a database migration error.

## 5. Confirm PostgreSQL and Cloudinary

Open `cult-records-db` in the Render Dashboard and confirm that its status is available. The Blueprint places the database and web service in the same workspace and Frankfurt region.

Open the web service's **Environment** page. `DATABASE_URL` should appear as a value linked from `cult-records-db`; it should not be a manually copied external database URL. The internal connection keeps application-to-database traffic on Render's private network.

The same page must contain `CLOUDINARY_URL` as a secret value. Open the deployed catalogue and confirm that product artwork uses URLs on `res.cloudinary.com`. Upload a temporary test image through the custom admin panel, confirm that it displays after saving, and then remove the temporary record or image through the normal admin controls.

Do not replace `DATABASE_URL` with the database's external URL unless the application is being run outside Render. External database access has separate TLS and access-control requirements.

## 6. Add the custom domain

The current Blueprint declares `cult.chriskalos.xyz`. After the Blueprint has been applied, open the web service and find **Settings** > **Custom Domains**. If the hostname is not already listed, select **Add Custom Domain**, enter `cult.chriskalos.xyz`, and save it.

Keep the Render page open because it shows the service's `onrender.com` hostname and the domain's verification status.

In Cloudflare:

1. Select the `chriskalos.xyz` zone.
2. Open **DNS** > **Records**.
3. Create a `CNAME` record.
4. Set **Name** to `cult`.
5. Set **Target** to `cult-records.onrender.com`.
6. Set **Proxy status** to **DNS only**, shown as a gray cloud.
7. Leave the TTL on **Auto**.
8. Remove any `AAAA` record for the exact hostname `cult.chriskalos.xyz`. Do not remove unrelated records for the rest of the domain.

Return to Render's **Custom Domains** section and select **Verify**. DNS changes can take several minutes to propagate. Once verification succeeds, wait for Render to report that the TLS certificate is issued and valid. Open `https://cult.chriskalos.xyz` and confirm that the page loads.

### Keep the record on DNS only

Keep the `cult` CNAME set to **DNS only**, shown as a gray cloud. Do not change it to **Proxied** after Render verifies the domain.

Cloudflare proxying was tested with this deployment after Render had verified the hostname and issued its certificate, but the proxied route did not fully work. The working configuration is:

- Type: `CNAME`
- Name: `cult`
- Target: `cult-records.onrender.com`
- Proxy status: **DNS only**
- TTL: **Auto**
- No `A` or `AAAA` record for the same hostname

With DNS-only routing, visitors still receive HTTPS directly through Render's managed certificate. Cloudflare continues to provide the domain's authoritative DNS, but its HTTP proxy and cache are not placed in front of the Render service.

If the custom domain does not work, confirm that the record is still set to **DNS only**, the CNAME points directly to the service's `onrender.com` hostname, Render shows the custom hostname as verified, and there is no conflicting `A` or `AAAA` record for `cult`. Do not turn proxying on for this deployment.

## 7. Enable Stripe sandbox checkout, if required

The catalogue and cart work without Stripe credentials, but starting a hosted checkout requires Stripe sandbox values.

1. Sign in to Stripe and select a sandbox rather than live mode.
2. Open the sandbox's API key page and copy its secret key. Use a test key beginning with `sk_test_`, or a restricted test key beginning with `rk_test_` that can create and retrieve Checkout Sessions.
3. In Render, open the `cult-records` web service and then **Environment**.
4. Set `STRIPE_SECRET_KEY` to the sandbox secret key and save the changes.
5. In Stripe's webhook area, create an HTTPS webhook endpoint at `https://cult.chriskalos.xyz/cart/stripe/webhook/`.
6. Subscribe the endpoint to:
   - `checkout.session.completed`
   - `checkout.session.async_payment_succeeded`
   - `checkout.session.expired`
7. Copy the webhook endpoint's signing secret, beginning with `whsec_`.
8. Return to Render's **Environment** page, set `STRIPE_WEBHOOK_SECRET`, and save the changes.
9. Allow the resulting deployment to finish, then perform a sandbox checkout using Stripe's documented test card details.

Never enter live Stripe keys in this deployment. The application deliberately rejects live secret keys.

## 8. Deploy future changes

The web service has automatic deployment enabled. A push to the linked GitHub `main` branch starts a new Render deployment. Pushing only to the Tangled mirror does not trigger Render.

For a normal update:

1. Merge or push the tested change to `main` on GitHub.
2. Open the Render service's **Events** page.
3. Wait for the new deployment to become **Live**.
4. Open the site and test the changed area.

If automatic deployment does not start, open the service's **Manual Deploy** menu and choose **Deploy latest commit**. Check the service's Git credentials under **Settings** if Render cannot read the repository.

Changes to `render.yaml` are also applied through the Blueprint. Review the Blueprint's proposed changes carefully before accepting destructive database or service changes.

The Tangled repository can continue to be maintained as a mirror, but both remotes should contain the same tested commit so neither copy falls behind.

## 9. Free-tier limitations

This setup is suitable for a demonstration or very lightly used project, but it is not durable production hosting.

- A free web service spins down after 15 minutes without incoming traffic. Its next request can take about a minute while it starts again.
- The web service filesystem remains ephemeral, but the application does not use it for production image storage.
- Images uploaded through the admin panel are stored in Cloudinary and remain available across restarts, idle spin-downs, and deployments.
- Version-controlled product artwork, HAM portraits, and the brand logo are delivered through Cloudinary. Their source copies remain in Git for reproducibility.
- PostgreSQL stores application records independently of the web service filesystem.
- A free PostgreSQL database has 1 GB of storage, no backups, and expires 30 days after creation.
- After database expiration, Render currently provides a 14-day grace period for upgrading before deleting the database and its data.
- A free web service cannot attach a persistent disk.
- Free usage is also subject to Render's monthly instance-hour, bandwidth, and build-minute allowances.

Cloudinary persistence does not protect PostgreSQL records. For a database that must survive beyond the demonstration period, upgrade PostgreSQL before its expiration date and arrange regular backups.

## 10. Final verification checklist

A fresh deployment is complete when all of the following are true:

- The Blueprint shows both `cult-records` and `cult-records-db`.
- The latest web-service deployment is **Live**.
- The home page returns successfully at the `onrender.com` address.
- The web service has a linked `DATABASE_URL`, a generated `SECRET_KEY`, and a secret `CLOUDINARY_URL`.
- Registration, sign-in, catalogue browsing, and the cart can write data to PostgreSQL.
- Static CSS and JavaScript load correctly, and all project images load from `res.cloudinary.com`.
- A new image uploaded through the custom admin panel remains visible after a service restart or idle spin-down.
- Render shows the custom domain as verified and its certificate as valid.
- The custom domain loads over HTTPS.
- The Cloudflare `cult` CNAME remains **DNS only**, shown as a gray cloud.
- If Stripe is enabled, sandbox checkout and the signed webhook both succeed.
- The database expiration date has been recorded by the person maintaining the deployment.
